Volver a implementar aplicaciones o módulos

Utilice la página Volver a implementar aplicaciones o módulos para implementar de nuevo una aplicación previamente implementada.

La página Volver a implementar aplicaciones o módulos contiene las siguientes opciones.

Ubicación

Ubicación del nuevo archivo de almacenamiento para la aplicación que está volviendo a implementar.

Las opciones siguientes especifican desde dónde se puede acceder al archivo del almacenamiento y si el archivo de almacenamiento es un archivo o un directorio.

Archivo empaquetado que se cargará en el servidor

El archivo de almacenamiento es un archivo que reside en la máquina cliente o al que se puede acceder desde la misma.

La máquina cliente es el host en el que está viendo la Consola de administración por medio de un navegador.

Directorio o archivo empaquetado local al que se puede acceder desde Servidor GlassFish

El archivo de almacenamiento es un archivo que reside en la máquina servidor o es una aplicación desempaquetada en un directorio de despliegue.

El equipo servidor es el sistema anfitrión que está ejecutando el servidor de administración del dominio de Servidor GlassFish.

Nombre de la aplicación

Campo de sólo lectura que muestra el nombre de la aplicación que está volviendo a implementar.

El nombre puede incluir un identificador de versión opcional, que va a continuación del nombre y está separado de él por dos puntos (:). Para obtener más información sobre las versiones de módulo y de aplicación, consulte Module and Application Versions de Oracle GlassFish Server 3.1 Application Deployment Guide.

Precompilar JSP

Si está seleccionada esta opción, los archivos JavaServer Pages ( JSP) se precompilarán. Si se deshabilita esta opción, los archivos JSP se compilarán en el tiempo de ejecución cuando se acceda a los mismos por primera vez. Esta opción se encuentra deshabilitada de forma predeterminada.

Ejecutar verificador

Si esta opción está seleccionada, los descriptores de implementación se verifican antes de la implementación. Si se produce un error en la verificación, no se realiza la implementación. El verificador examinará la estructura y el contenido de los descriptores de implementación. La verificación de aplicaciones de gran tamaño tiende a llevar mucho tiempo. Esta opción se deshabilita por defecto.

Los paquetes de verificación se deben haber instalado desde la Herramienta de actualización o se registrará una advertencia y se hará caso omiso de esta opción.

Java Web Start

Si esta opción está seleccionada, se permite el acceso a Java Web Start para un módulo de cliente de aplicaciones. Esta opción se deshabilita por defecto.

Estado de conservación

Esta opción controla si las sesiones web, instancias SFSB y temporizadores EJB creados de forma persistente se conservan entre reimplementaciones.

Esta opción se deshabilita por defecto. Esta opción sólo se admite en la instancia de servidor predeterminada, denominada server. No se admite para ningún otro destino, y se hace caso omiso de ella.

Algunos cambios efectuados en una aplicación entre reimplementaciones impiden que esta opción funcione correctamente. Por ejemplo, no cambie el conjunto de variables de instancia en la clase de bean SFSB.

Para las aplicaciones web, esta opción solo es aplicable si, en el archivo glassfish-web-app.xml, el atributo persistence-type del elemento session-manager es file.

Para instancias de bean de sesión con estado, el tipo de persistencia sin alta disponibilidad se establece en el servidor (con la opción Tipo de persistencia SFSB) y se debe configurar como file, que es el valor predeterminado y recomendado.

Si una sesión web, instancia SFSB o temporizador EJB activo no se conserva o restablece, ninguno de ellos estará disponible cuando se haya completado la reimplementación. Sin embargo, la reimplementación proseguirá y se registrará una advertencia.

Para conservar datos de estado activo, Servidor GlassFish serializa los datos y los guarda en la memoria. Para restaurar los datos, el cargador de clases de la aplicación que se acaba de volver a implementar deserializa los datos guardados anteriormente.

Conservación de recursos de ámbito de aplicación

Si se selecciona, conserva todos los recursos de ámbito restringido a la aplicación y los restaura durante la reimplementación. Esta opción no está activada de forma predeterminada.
