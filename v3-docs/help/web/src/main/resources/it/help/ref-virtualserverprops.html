Proprietà specifiche dei server virtuali

Per un server virtuale sono disponibili le seguenti proprietà aggiuntive.

sso-max-inactive-seconds

Specifica il numero di secondi al termine dei quali è possibile eliminare un record SSO dell'utente se non viene ricevuta alcuna attività del client. Poiché il sistema SSO viene utilizzato con più applicazioni sullo stesso server, l'accesso a una qualsiasi delle applicazioni mantiene attivo il record di autorizzazioni SSO.

Il valore predefinito è 300 secondi (5 minuti). Un valore superiore consente di ottenere una persistenza SSO maggiore per gli utenti, ma causa un maggior consumo di memoria nel server.

sso-reap-interval-seconds

Specifica il numero di secondi che intercorrono tra le eliminazioni di record SSO scaduti.

Il valore predefinito è 60.

ssoCookieSecure

Imposta l'attributo Secure di qualsiasi cookie JSESSIONIDSSO associato alle applicazioni Web distribuite in questo server virtuale. Questa proprietà è valida solo se il sistema SSO è abilitato. Di seguito sono riportati i valori validi:

true

Imposta Secure su true.

false

Imposta Secure su false.

dynamic

Il cookie JSESSIONIDSSO eredita l'impostazione Secure dalla prima sessione in cui viene utilizzata l'autorizzazione SSO.

Per impostare l'attributo Secure di un cookie JSESSIONID, utilizzare la proprietà cookieSecure cookie-properties nel file sun-web.xml. Per i dettagli, vedere cookie-properties in Oracle GlassFish Server 3.1 Application Deployment Guide.

setCacheControl

Specifica un elenco separato da virgole di direttive di risposta Cache-Control. Per un elenco di direttive valide, vedere la sezione 14.9 del documento all'indirizzo http://www.ietf.org/rfc/rfc2616.txt.

accessLogBufferSize

Specifica le dimensioni in byte del buffer in cui vengono memorizzate le chiamate del log di accesso. Se il valore è inferiore a 5120, viene visualizzato un messaggio e il valore viene impostato su 5120. Il valore predefinito è 32768.

accessLogWriterInterval

Specifica il numero di secondi precedenti alla scrittura del log sul disco. Il log di accesso viene scritto quando il buffer è pieno o allo scadere dell'intervallo. Se il valore è pari a 0, il buffer verrà sempre scritto, anche se non è pieno. Ogni volta che si accede al server, i messaggi del log vengono pertanto memorizzati direttamente nel file. Il valore predefinito è 300.

allowRemoteAddress

Specifica un elenco separato da virgole di modelli di espressioni regolari con cui viene confrontato l'indirizzo IP del client remoto. Se la proprietà viene specificata, l'indirizzo remoto deve corrispondere a un modello affinché venga accettata la richiesta. Se la proprietà non viene specificata, vengono accettate tutte le richieste a meno che l'indirizzo remoto non corrisponda al modello denyRemoteAddress.

denyRemoteAddress

Specifica un elenco separato da virgole di modelli di espressioni regolari con cui viene confrontato l'indirizzo IP del client remoto. Se la proprietà viene specificata, l'indirizzo remoto non deve corrispondere a un modello affinché venga accettata la richiesta. Se la proprietà non viene specificata, l'accettazione della richiesta viene controllata dalla proprietà allowRemoteAddress.

allowRemoteHost

Specifica un elenco separato da virgole di modelli di espressioni regolari con cui il nome host del client remoto (restituito da [java.net.]Socket.getInetAddress().getHostName() ) viene confrontato. Se la proprietà viene specificata, il nome host remoto deve corrispondere a un modello affinché venga accettata la richiesta. Se la proprietà non viene specificata, vengono accettate tutte le richieste a meno che il nome host remoto non corrisponda al modello denyRemoteHost.

denyRemoteHost

Specifica un elenco separato da virgole di modelli di espressioni regolari con cui il nome host del client remoto (restituito da [java.net.]Socket.getInetAddress().getHostName() ) viene confrontato. Se la proprietà viene specificata, il nome host remoto non deve corrispondere a un modello affinché venga accettata la richiesta. Se la proprietà non viene specificata, l'accettazione della richiesta viene controllata dalla proprietà allowRemoteHost.

L'impostazione di questa proprietà non ha effetto se l'accesso al dominio GlassFish Server viene eseguito tramite un listener di rete con l'opzione JK Listener abilitata.

authRealm

Specifica il nome di un settore di autenticazione che sovrascrive il settore predefinito dell'istanza server per le applicazioni Web standalone distribuite nel server virtuale. Un settore definito nel file web.xml di un'applicazione Web standalone ha la priorità sul settore del server virtuale. Per ulteriori informazioni, vedere Realms.

securePagesWithPragma

Impostare questa proprietà su false per assicurare che per tutte le applicazioni Web su questo server virtuale i download di file tramite SSL funzionino correttamente in Internet Explorer.

In singole applicazioni Web è possibile sovrascrivere questa impostazione utilizzando l'elemento sun-web-app del file sun-web.xml. Per informazioni dettagliate, vedere glassfish-web-app in Oracle GlassFish Server 3.1 Application Deployment Guide.

Il valore predefinito è true.

alternatedocroot_n

Specifica una root alternativa di documenti (docroot), dove n è un numero intero positivo che consente di specificarne più di una. Le docroot alternative consentono alle applicazioni Web di eseguire richieste per determinate risorse dall'esterno della rispettiva docroot, sulla base della eventuale corrispondenza tra tali richieste e uno o più modelli URI delle docroot alternative dell'applicazione Web.

Se una richiesta corrisponde a un modello URI della docroot alternativa, questa viene mappata sulla docroot alternativa mediante l'aggiunta dell'URI della richiesta (senza la root di contesto dell'applicazione Web) alla posizione fisica della docroot alternativa (directory). Se una richiesta corrisponde a più modelli URI, la docroot alternativa viene determinata in base al seguente ordine di priorità:

  • Corrispondenza esatta

  • Corrispondenza tra percorsi di lunghezza massima

  • Corrispondenza tra estensioni

Ad esempio, le seguenti proprietà definiscono tre docroot alternative. Il modello URI della prima docroot alternativa utilizza una corrispondenza esatta, mentre i modelli URI della seconda e della terza docroot alternativa utilizzano rispettivamente corrispondenze tra estensioni e prefissi di percorsi di lunghezza massima.

<property name="alternatedocroot_1" 
   value="from=/my.jpg dir=/srv/images/jpg"/>
<property name="alternatedocroot_2" 
   value="from=*.jpg dir=/srv/images/jpg"/>
<property name="alternatedocroot_3" 
   value="from=/jpg/* dir=/src/images"/>

Il value di ogni docroot alternativa è costituito da due componenti, from, che specifica il modello URI della docroot alternativa, e dir, che specifica la posizione fisica della docroot alternativa (directory). Nel componente dir è consentito l'uso di spazi.

In singole applicazioni Web è possibile sovrascrivere questa impostazione utilizzando l'elemento sun-web-app del file sun-web.xml. Per informazioni dettagliate, vedere glassfish-web-app in Oracle GlassFish Server 3.1 Application Deployment Guide.

contextXmlDefault

Specifica la posizione, relativamente a domain-dir, del file context.xml di questo server virtuale, se disponibile. Per ulteriori informazioni sul file context.xml, vedere The Context Container.

allowLinking

Se true, le risorse che rappresentano collegamenti simbolici verranno eseguite per tutte le applicazioni Web distribuite in questo server virtuale. In singole applicazioni Web è possibile sovrascrivere questa impostazione utilizzando la proprietà sun-web-app allowLinking nel file sun-web.xml:

<sun-web-app>
   <property name="allowLinking" 
      value="{true|false}"/>
</sun-web-app>

Per informazioni dettagliate, vedere glassfish-web-app in Oracle GlassFish Server 3.1 Application Deployment Guide.

Il valore predefinito è false.

Avvertenza - Se si imposta questa proprietà su true nei sistemi Windows, il codice sorgente JSP sarà visibile.

send-error_n

Specifica i mapping di pagine di errore personalizzate per il server virtuale, che sono ereditati da tutte le applicazioni Web distribuite nel server virtuale. Un'applicazione Web può sovrascrivere questi mapping di pagine di errore personalizzate nel rispettivo descrittore di distribuzione web.xml. Il valore di ogni proprietà send-error_ n è costituito da tre componenti, che possono essere specificati in qualsiasi ordine:

  • Il primo componente, code, specifica il codice dello stato di risposta HTTP a tre cifre in base al quale la pagina di errore personalizzata dovrebbe essere restituita nella risposta.

  • Il secondo componente, path, specifica il percorso del file system completo o relativo della pagina di errore personalizzata. Un percorso del file system relativo viene interpretato come relativo rispetto alla directory domain-dir/config.

  • Il terzo componente, reason, è facoltativo e specifica il testo della stringa esplicativa (quale Unauthorized o Forbidden) da restituire.

Ad esempio:

<property name="send-error_1"
   value="code=401 path=/myhost/401.html reason=MY-401-REASON"/>

Sulla base di questa definizione di proprietà di esempio, il contenuto di /myhost/401.html viene restituito con 401 risposte, insieme alla seguente riga di risposta:

HTTP/1.1 401 MY-401-REASON
redirect_n

Specifica di considerare la richiesta indirizzata a un URL precedente come una richiesta per un nuovo URL. Queste proprietà sono ereditate da tutte le applicazioni Web distribuite nel server virtuale. Il valore di ogni proprietà redirect_ n è costituito da due componenti, che possono essere specificati in qualsiasi ordine:

  • Il primo componente, from, specifica il prefisso dell'URI richiesto per il quale individuare corrispondenze.

  • Il secondo componente, url-prefix, specifica il nuovo prefisso URL da restituire al client. Il prefisso from viene semplicemente sostituito da questo prefisso URL.

Ad esempio:

<property name="redirect_1" 
  value="from=/dummy url-prefix=http://etude"/>
valve_n

Specifica il nome completo di una classe Valve personalizzata, dove n è un numero intero positivo che consente di specificarne più di una. La classe Valve deve implementare l'interfaccia org.apache.catalina.Valve da Tomcat o da release di GlassFish Server precedenti, oppure l'interfaccia org.glassfish.web.valve.GlassFishValve dal release corrente di GlassFish Server. Ad esempio:

<property name="valve_1"
value="org.glassfish.extension.Valve"/>

È possibile impostare questa proprietà per un'applicazione Web specifica. Per informazioni dettagliate, vedere glassfish-web-app in Oracle GlassFish Server 3.1 Application Deployment Guide.

listener_n

Specifica il nome completo della classe di un listener Catalina personalizzato, dove n è un numero intero positivo che consente di specificarne più di uno. La classe di listener deve implementare l'interfaccia org.apache.catalina.ContainerListener o org.apache.catalina.LifecycleListener . Ad esempio:

<property name="listener_1"
value="org.glassfish.extension.MyLifecycleListener"/>

È possibile impostare questa proprietà per un'applicazione Web specifica. Per informazioni dettagliate, vedere glassfish-web-app in Oracle GlassFish Server 3.1 Application Deployment Guide.

errorReportValve

Specifica il nome completo di una classe Valve personalizzata che elabora pagine di errore predefinite per applicazioni in questo server virtuale. Specificare una stringa vuota per disabilitare la modalità di elaborazione di pagine di errore predefinite per questo server virtuale.
