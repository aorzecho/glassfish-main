Créer un serveur virtuel

  1. Dans l’arborescence de navigation, développez le nœud Configuration.
  2. Sous le nœud Configuration, sélectionnez le nœud Serveurs virtuels.

    La page Serveurs virtuels s’ouvre.

  3. Dans la page Serveurs virtuels, cliquez sur Nouveau.

    La page Nouveau serveur virtuel s’ouvre.

  4. Dans le champ ID, entrez une valeur identifiant le serveur virtuel en interne.

    Cette valeur n’est pas visible pour les clients HTTP. Les noms d’hôtes visibles des clients HTTP doivent être spécifiés dans le champ Hôtes.

  5. Dans le champ Hôtes, entrez les noms d’hôte ou les adresses IP de la machine sur laquelle le serveur s’exécute.

    Utilisez des noms d’hôte réels ou virtuels enregistrés dans le serveur DNS pour votre réseau (et, sous UNIX, dans votre fichier /etc/hosts).

    La valeur par défaut est la valeur de propriété système ${com.sun.aas.hostName}.

  6. Sélectionnez une option État pour le nouveau serveur virtuel.

    Le serveur virtuel peut être défini sur On (c’est-à-dire actif), Off (inactif) ou Disabled (désactivé). La valeur par défaut est On.

  7. Sélectionnez une option SSO pour le nouveau serveur virtuel.

    Par défaut, un serveur virtuel hérite du paramètre de connexion unique (SSO) du service HTTP. Sélectionnez Activé ou Désactivé pour annuler le paramètre du service HTTP en activant ou en désactivant la connexion unique pour ce serveur virtuel.

    La connexion unique d’une application à l’autre dans GlassFish Server est prise en charge par les servlets et les fichiers JSP. Cette fonction permet à plusieurs applications exigeant les mêmes informations de connexion utilisateur de partager ces dernières au lieu de contraindre l’utilisateur à s’authentifier séparément pour chaque application.

  8. Dans la liste Auditeurs de réseau, sélectionnez un ou plusieurs auditeurs de réseau, ou laissez le champ vide.
  9. Dans la liste déroulante Module Web par défaut, sélectionnez le module devant répondre à toutes les requêtes qui ne peuvent pas être mappées vers les autres modules Web déployés sur le serveur virtuel.

    Si aucun module Web par défaut n’est spécifié, le module Web qui possède une racine contexte vide est utilisé. S’il n’existe aucun module Web avec une racine contexte vide, un module Web par défaut du système est créé et utilisé.

  10. Dans le champ Fichier journal, saisissez le nom de chemin du fichier où les messages de journalisation provenant de ce serveur virtuel doivent apparaître.

    Par défaut, des messages de journalisation sont envoyés au journal du serveur par défaut, domain-dir/logs/server.log.

  11. Dans le champ Docroot, entrez le chemin absolu du répertoire racine de documents sur le serveur.

    Si ce champ n’est pas renseigné, domain-dir/docroot est utilisé.

  12. Sélectionnez l’option Journalisation de l’accès pour activer ou désactiver la journalisation des accès.

    Sinon, vous pouvez également indiquer si la journalisation des accès est héritée du service HTTP. Par défaut, elle est héritée du service HTTP.

  13. Dans le champ Répertoire, entrez le chemin absolu des journaux d’accès du serveur.

    La valeur par défaut est domain-dir/logs/access.

  14. (Facultatif) Dans la section Autres propriétés, spécifiez les autres propriétés du serveur virtuel.

    Pour obtenir des informations sur les propriétés disponibles, consultez Propriétés spécifiques aux serveurs virtuels.

    Pour ajouter une propriété, cliquez sur le bouton Ajouter propriété. Sur la ligne vide qui apparaît, saisissez le nom de la propriété dans le champ Nom et la valeur de la propriété dans le champ Valeur.

  15. Cliquez sur OK.
